# 高性能实时渲染 Markdown 桌面应用：Codex 实施计划

> 版本：1.0  
> 目标平台：macOS 
> 核心技术：Tauri v2、Rust、Svelte、TypeScript、CodeMirror 6、Lezer  
> 最高原则：实时渲染、高性能、省电

## 0. 给 Codex 的执行指令

你要构建一个类似 Typora 的单窗口实时渲染 Markdown 桌面应用。底层文档始终是标准 Markdown 纯文本；普通语法在编辑器中直接呈现排版效果，光标进入相关语法区域时恢复源码；复杂语法使用独立渲染块。

必须严格按照本计划循序实施：

1. 一次只实施一个阶段，不得提前实现后续功能。
2. 每个阶段完成后，必须运行该阶段规定的测试和性能验证。
3. 必须记录真实测量数据；严禁填写未经测量的“预计提升百分比”。
4. 阶段门禁未通过时，停止扩展功能，先分析和解决当前问题。
5. 不得为了开发方便破坏实时渲染、高性能、省电三个最高原则。
6. 不进行与当前阶段无关的重构，不提前建立通用插件生态、云同步或协作系统。
7. 任何新增的定时器、后台线程、文件监听、缓存或依赖，都必须说明其触发条件、空闲成本和释放方式。
8. 每阶段结束时提交一份检查点报告，至少包含：修改文件、完成能力、测试结果、性能结果、遗留问题和下一阶段风险。

首次执行本计划时，只实施“阶段 0：工程骨架与测量基线”。完成并通过门禁后停止，等待确认，不得自动进入阶段 1。

## 1. 项目背景与产品定位

### 1.1 产品目标

构建一个轻量、离线优先、跨平台的 Markdown 桌面编辑器，重点提供：

- 类 Typora 的单窗口实时渲染体验；
- 输入时低延迟、滚动流畅；
- 空闲时接近无业务 CPU 活动；
- 对大文档进行增量分析和视口渲染；
- 数学公式、Mermaid、Git 等高级能力默认关闭，按需启用；
- 外部文件变化时明确提示，不静默覆盖；
- 文件始终保持标准 Markdown，可使用其他工具正常打开和编辑。

### 1.2 实时渲染的准确边界

本项目不是传统富文本编辑器，也不是左右分栏预览器，而是 WYSIWYM 风格的 Markdown 编辑器：

- 标题、粗体、斜体、删除线、引用、列表等直接显示排版效果；
- 非活动区域隐藏部分 Markdown 标记；
- 光标或选区进入相关语法节点时显示原始 Markdown；
- 代码块、表格、公式、图表等复杂内容可以显示为独立块；
- 用户点击复杂块后，切换到该块的 Markdown 源码编辑状态；
- 文档的唯一可持久化格式始终是 Markdown 文本。

### 1.3 第一阶段明确不做的内容

- 第三方插件市场和公共插件 API；
- WASM、QuickJS、Lua 等插件沙盒；
- 云同步、多人协作和 CRDT；
- 移动端与 Linux 正式支持；
- 所有 Typora 功能的完整兼容；
- PDF、Word 等复杂导出；
- 全文知识库、双向链接和数据库式笔记能力；
- 自研 GPU 渲染引擎；
- Electron 版本。

## 2. 最高原则与不可破坏的工程约束

### 2.1 实时渲染

- 普通输入的视觉反馈必须在 CodeMirror 当前事务或紧接着的浏览器帧内完成。
- 输入主路径不得依赖 Rust IPC、全量 Markdown 解析、磁盘 I/O 或网络请求。
- 标题、强调、链接等基础样式使用 CodeMirror 语法树与 Decorations 实现。
- Mermaid、数学公式等重型渲染不进入同步输入路径，必须延迟、可取消并检查文档版本。
- 中文、日文等 IME composition 期间，不得破坏正在组合的文本和光标。

### 2.2 高性能

- 不在每次按键时把全文发送给 Rust。
- 不在每次按键时遍历全文、计算每行 Hash 或重新生成完整 HTML。
- 优先使用 CodeMirror 的 `EditorState`、`ChangeSet`、Lezer 增量语法树、`visibleRanges` 和 Decorations。
- Svelte 只管理应用壳，禁止把完整文档内容放入 Svelte 高频响应式状态。
- 异步渲染必须携带文档 revision；旧结果不得覆盖新状态。
- 缓存必须有明确的容量上限，优先按估算字节数限制，而不是只限制条目数。

### 2.3 省电

- 系统以事件驱动为主；没有变化时不做业务计算。
- 禁止为了“流畅”建立持续 60 FPS 渲染循环。
- 禁止没有必要的 `setInterval`、轮询式性能监控和定期全文扫描。
- 防抖使用变脏后创建的一次性 deadline；任务执行且状态恢复干净后，不继续唤醒。
- 窗口隐藏时暂停或延后复杂渲染、索引和 Git Push，但不能无条件停止关键文件冲突检测。
- 不得关闭 WebView 自带的后台定时器节流。
- 功能关闭时，其后台任务、Worker、事件监听器和缓存也必须停止或释放。

### 2.4 数据安全是硬约束

性能优化不能以丢失内容、静默覆盖或破坏 Markdown 文件为代价：

- 文件保存必须采用安全的原子替换策略；
- 自己的保存事件与外部修改事件必须区分；
- 外部版本与内存版本冲突时不得自动覆盖；
- Git 操作不得自动强推、不得静默丢弃工作区变更；
- 异常退出后应有受控的恢复能力。

## 3. 技术选型与职责边界

| 层次 | 技术 | 职责 | 禁止承担的职责 |
|---|---|---|---|
| 桌面壳 | Tauri v2 | 窗口、菜单、权限、IPC、打包 | 每次输入的实时渲染 |
| 应用 UI | Svelte + TypeScript + Vite | 标签页、文件树、设置、弹窗、状态栏 | 持有全文响应式副本 |
| 编辑内核 | CodeMirror 6 | 文档状态、输入、选区、撤销、视口 | 磁盘写入和 Git |
| 增量语法 | `@codemirror/lang-markdown` + Lezer | 实时语法树、可视区域分析 | 最终导出 HTML 的唯一依据 |
| 实时渲染 | Decorations + WidgetType | 行内样式、标记隐藏、块级渲染 | 直接修改 CodeMirror 管理的 DOM |
| Rust 核心 | Rust + Tauri commands | 文件、配置、监听、恢复、系统能力 | 输入时全量解析并返回 HTML |
| 完整 Markdown 处理 | `pulldown-cmark`，后期引入 | 导出、完整预览、一致性验证 | 编辑器级增量解析 |
| 文件监听 | `notify` | 已打开文件的外部变化检测 | 默认递归监听整个工作区 |
| Git | 系统 Git CLI，功能默认关闭 | Commit、Push、状态检测 | 每次保存同步执行 Git |

### 3.1 文档状态原则

打开文档后：

- 当前编辑会话的权威状态是 CodeMirror `EditorState.doc`；
- Rust 保存磁盘版本、最后保存 Hash、文件元数据和文档标识；
- MVP 保存时发送完整快照，不在每次按键时同步 Rust Rope；
- 只有大文件基准证明快照序列化成为实际瓶颈后，才允许引入版本化增量同步；
- 若引入 Rust 文档镜像，必须采用 `document_id + base_revision + next_revision` 协议，并能在版本不一致时全量重同步。

### 3.2 输入主路径

```text
键盘或 IME 输入
  → CodeMirror Transaction
  → Lezer 增量语法树更新
  → 当前视口 Decorations 更新
  → 浏览器绘制
```

这条路径不得经过：

- Tauri IPC；
- Rust Markdown 解析；
- 文件系统；
- Git；
- 网络；
- 完整文档字符串复制；
- 重型公式或图表渲染。

### 3.3 保存路径

```text
CodeMirror 文档快照
  → UTF-8 编码
  → Tauri 原始 Buffer 或受控命令
  → Rust 原子保存
  → 更新 saved_revision / content_hash
  → 过滤本应用产生的 watcher 事件
```

## 5. 性能与功耗测量体系

### 5.1 固定测试文档

建立可重复生成的测试夹具：

| 夹具 | 建议规模 | 内容 |
|---|---:|---|
| `basic.md` | 20–50 KB | 常用 Markdown 语法 |
| `medium.md` | 约 1 MB | 段落、列表、引用、代码块混合 |
| `large.md` | 约 10 MB | 长文档和大量行 |
| `pathological.md` | 1–5 MB | 深层列表、未闭合围栏、长行、HTML Block |
| `widgets.md` | 可配置 | 大量表格、公式、Mermaid 块 |
| `ime.md` | 小型 | 中文、日文、Emoji、组合字符、RTL 样例 |

夹具必须可重复生成，不能把巨大随机文件直接提交进仓库。

### 5.2 初始性能门禁

以下是首轮工程目标，不是营销承诺。测试报告必须记录硬件、系统、Debug/Release 模式和测量方法。

| 场景 | 初始门禁 |
|---|---|
| `basic.md` 连续输入 | 按键到视图更新 p95 < 8 ms，p99 < 16.7 ms |
| `medium.md` 连续输入 | p95 < 16.7 ms，不出现连续长任务 |
| `large.md` 源码/降级模式 | 普通输入 p95 < 33 ms，应用可正常保存和退出 |
| 普通滚动 | 无持续明显掉帧，复杂块不得同步阻塞滚动 |
| 前台静置 60 秒 | 应用自有逻辑无重复工作；平均 CPU 目标不高于测试机 0.5% |
| 后台静置 60 秒 | 无应用自建高频唤醒；没有启用功能的后台任务不得运行 |
| 重复滚完整文档 | 内存不能随滚动次数持续线性增长 |
| 功能关闭 | 不加载对应重型模块，不创建 Worker、定时器和模块缓存 |

如果 WebView 或系统本身造成指标波动，报告中必须区分“应用逻辑成本”和“平台基线”，不得篡改数据使其通过。

### 5.3 测量要求

- 开发构建用于调试，正式性能数据必须来自 Release 构建；
- 使用 `performance.mark/measure` 记录编辑事务和 Decoration 构建时间；
- 监控 Long Task，但不能让监控本身成为生产环境长期轮询；
- macOS 使用 Instruments 的 Time Profiler、Energy Log，必要时结合系统工具；
- Windows 使用 Windows Performance Recorder/Analyzer 或等价 ETW 工具；
- 每阶段性能报告保存到 `docs/performance/`；
- 若某个优化没有基准数据证明有效，不得仅凭直觉加入复杂缓存或双文档模型。

## 6. 分阶段实施计划

## 阶段 0：工程骨架与测量基线

### 目标

建立最小可运行工程和性能测量基础，不实现 Typora 式标记隐藏，不引入文件监听、Git、KaTeX 或 Mermaid。

### 实施内容

1. 创建 Tauri v2 + Vite + Svelte + TypeScript 工程。
2. 启用 TypeScript 严格模式。
3. 集成 CodeMirror 6 和 Markdown language support。
4. Svelte 中只挂载一个编辑器容器，不双向绑定完整文档。
5. 建立最小主题和单窗口布局。
6. 建立 Rust、TypeScript 的统一错误记录方式，生产模式禁止高频 console 输出。
7. 建立基础测试命令：格式化、类型检查、单元测试、Rust 测试、Clippy。
8. 建立测试夹具生成器和最小性能采样工具。
9. 记录空编辑器、`basic.md`、`medium.md` 的启动、输入和静置基线。
10. 新建 ADR：
    - `001-editor-state-authority.md`；
    - `002-input-path-and-ipc-boundary.md`；
    - `003-performance-measurement.md`。

### 明确禁止

- 不实现实时标记隐藏；
- 不加入 `setInterval` 性能轮询；
- 不让 Rust 接收每次输入；
- 不引入 Ropey、pulldown-cmark、notify、git2；
- 不做复杂 UI。

### 验证

- 编辑、撤销、重做、选区、粘贴正常；
- 中文 IME 可完成输入；
- CodeMirror 文档变化不会导致 Svelte 根组件高频重渲染；
- Rust 端在输入过程中没有 IPC 调用；
- 所有静态检查和测试通过；
- 生成 `docs/checkpoints/phase-0.md` 和性能基线。

### 阶段门禁

- Release 构建可运行；
- 无应用自建持续渲染循环；
- 静置时不存在由本项目代码产生的高频任务；
- 性能测量能够稳定重复执行；
- 未通过门禁时不得进入阶段 1。

## 阶段 1：基础 Live Preview 垂直切片

### 目标

验证项目最大技术风险：在 CodeMirror 6 内实现可靠、低延迟、支持 IME 的 Typora 式基础实时渲染。

### 首批语法

- H1–H6 标题；
- 粗体、斜体、粗斜体；
- 删除线；
- 行内代码；
- 普通链接；
- 引用；
- 有序、无序列表。

### 实施要求

1. 使用 Lezer Markdown 语法树定位节点，不用正则表达式扫描全文。
2. 只针对 `visibleRanges` 和必要缓冲区构建视图装饰。
3. 使用 `Decoration.mark` 设置内容样式。
4. 使用 `Decoration.replace` 隐藏 Markdown 标记，禁止使用 `width: 0` 作为核心方案。
5. 光标、选区进入语法节点或当前块时显示源码标记。
6. composition 期间冻结会影响当前组合区域的装饰重建；组合结束后再安全刷新。
7. Decorations 只在 `docChanged`、`selectionSet`、`viewportChanged` 或语法树继续完成时更新。
8. 保持复制内容为原始 Markdown，而不是渲染后的文本。
9. 不直接修改 `cm-content` 下的 DOM。
10. 不增加 Rust 解析和 IPC。

### 测试

- 每种语法的节点识别与 Decoration 范围单元测试；
- 光标进入、离开节点时标记显隐测试；
- 多选区、撤销、重做、复制、粘贴测试；
- 未闭合 `**`、反引号、链接等错误输入测试；
- 中文 IME、Emoji、组合字符测试；
- 快速连续输入和删除测试；
- `basic.md` 与 `medium.md` 性能测试。

### 阶段门禁

- 基础语法不需要 Rust IPC 即可实时渲染；
- 不破坏 IME、选区和撤销栈；
- 达到第 5 节中对应的输入性能门禁；
- Decorations 构建时间不会随不可见文档长度线性恶化；
- 生成 `docs/checkpoints/phase-1.md`；
- 若 IME 或光标行为不可靠，停止，不得用复杂块掩盖问题。

## 阶段 2：文件打开、保存与外部修改保护

### 目标

把编辑器变成安全可用的本地文件应用，同时不让文件 I/O 进入输入主路径。

### 实施内容

1. 使用 Tauri 权限明确的命令打开 Markdown 文件。
2. MVP 支持 UTF-8、UTF-8 BOM，并保留原始换行符风格。
3. 建立 `DocumentSession`：
   - `document_id`；
   - 路径；
   - `edit_revision`；
   - `saved_revision`；
   - 最后磁盘 Hash；
   - 文件元数据。
4. 先完成手动保存，再加入可配置的空闲自动保存。
5. 保存使用同目录临时文件与原子替换，正确处理失败和权限错误。
6. 保存成功后才更新 `saved_revision` 和磁盘 Hash。
7. 使用 `notify` 非递归监控所有已打开文件的父目录，相同目录共享 watcher。
8. 文件事件先按路径过滤，再防抖合并，最后验证 `mtime/size/hash`。
9. 本应用保存产生的事件通过保存 generation 和 Hash 抑制。
10. 确认外部内容变化后显示强提示：
    - 重新加载；
    - 保留当前版本；
    - 查看差异（可以先作为后续入口）。
11. 不自动合并，不引入 CRDT。

### 数据安全测试

- 原地写入；
- 临时文件替换式写入；
- 外部删除、重命名；
- 文件只读和磁盘写入失败；
- 自己保存不会触发外部冲突提示；
- 本地未保存修改与外部修改同时发生；
- Windows 文件占用与 macOS 原子替换差异；
- 应用关闭前的未保存提示。

### 性能与功耗验证

- 输入期间没有磁盘写入；
- 自动保存只在文档变脏后创建一次性 deadline；
- 干净状态不周期性保存或计算全文 Hash；
- watcher 只覆盖已打开文件相关目录，不递归整个工作区；
- 保存大文件不会长期阻塞 UI。

### 阶段门禁

- 所有失败路径都不会丢失内存中的用户内容；
- 外部变化不会静默覆盖；
- 手动保存与自动保存均通过集成测试；
- 生成 `docs/checkpoints/phase-2.md`。

---

## 阶段 3：大文档与视口性能

### 目标

在增加复杂渲染能力前，先证明基础编辑架构不会因为文档体积增长而失控。

### 实施内容

1. 使用 `medium.md`、`large.md`、`pathological.md` 建立性能曲线。
2. 检查 Decoration 是否仅构建可视范围及缓冲区。
3. 检查语法树后台解析是否会抢占输入帧。
4. 建立降级等级，但阈值必须根据实测决定：
   - 完整 Live Preview；
   - 只渲染视口内基础语法；
   - 暂停复杂块；
   - 超大文件源码模式。
5. 降级必须可见且可由用户覆盖，不能悄悄改变文件内容。
6. 验证 CodeMirror 自带视口管理后，再决定是否需要额外 DOM 回收；不得预先实现第二套虚拟滚动。
7. 测量保存快照的 UTF-8 编码与 IPC 开销。

### 增量持久化的启用条件

只有满足以下任一条件才实施 Rust 文档镜像：

- 大文件快照使 UI 出现超过门禁的长任务；
- 自动保存的全文编码和传输成为明确 CPU/内存热点；
- 性能剖析能够证明增量协议收益大于双状态复杂度。

若启用，必须实现：

- 版本化 ChangeSet 批量传输；
- 明确的 Unicode 位置语义；
- Rust ACK；
- 乱序和重复检测；
- revision 不匹配时全量重同步；
- 保存必须等待目标 revision 已同步；
- 协议级单元测试和随机编辑序列测试。

### 阶段门禁

- `large.md` 至少能在受控降级模式下可靠输入、保存和关闭；
- 快速滚动不会造成持续线性内存增长；
- 空闲 CPU 不因大文档而持续活跃；
- 没有性能证据时保持简单快照架构；
- 生成 `docs/checkpoints/phase-3.md`。

## 阶段 4：基础块级渲染

### 目标

在已验证的编辑器基础上加入块级能力，但不引入 KaTeX 和 Mermaid。

### 首批块

- 围栏代码块；
- 简单表格；
- 任务列表；
- 分隔线；
- 可选的图片占位和安全预览。

### 实施要求

1. 使用 CodeMirror Block Widget 或 replacing decoration，不创建独立预览面板。
2. 当前块获得光标时切换为可编辑源码状态。
3. 重型块在输入停止后的单次延迟任务中渲染，建议起点 100–200ms，最终以实测调整。
4. 每个异步任务携带 `document_id + revision + source_range/hash`。
5. 文档变化后取消或丢弃过期渲染结果。
6. 只实例化视口和有限缓冲区内的复杂块。
7. 缓存按估算字节数限制，并能在文档关闭或功能关闭时释放。
8. 代码语言高亮包按语言懒加载，不一次加载所有语言。

### 验证

- 快速滚动不会持续创建无法释放的 Widget；
- 块高度变化不会导致严重滚动跳动；
- 快速编辑时旧 Widget 不会覆盖新源码；
- 当前块源码与渲染块切换稳定；
- 100、500、1000 个复杂块的压力测试；
- 功能关闭后相关缓存和监听清空。

### 阶段门禁

- 块级能力不影响普通段落的输入性能门禁；
- 不存在异步结果竞态；
- 内存随重复滚动保持有界；
- 生成 `docs/checkpoints/phase-4.md`。

---

## 阶段 5：内置可选模块系统

### 目标

以“资源洁癖”为目的建立轻量内置模块机制，不建立第三方生态。

### 模块契约

每个模块至少定义：

```text
detect → activate → render → suspend → dispose
```

并明确：

- 配置是否启用；
- 何时加载代码；
- 创建了哪些 Worker、事件监听器、缓存和 DOM；
- 如何取消任务；
- 如何释放实例资源；
- 文档或应用关闭时如何清理。

### 实施顺序

1. 先完成模块注册、状态切换和资源审计，不加载真实重型库。
2. KaTeX 作为第一个真实模块。
3. KaTeX 验证通过后再加入 Mermaid。
4. JS、CSS、字体全部本地打包，通过动态 `import()` 延迟加载。
5. 禁止从 CDN 注入脚本。
6. 禁止宣称动态导入后的模块代码可以被 JavaScript 引擎彻底卸载；`dispose` 只承诺清理实例、DOM、Worker、监听器和缓存。
7. Cargo Features 仅用于构建不同发行版本，不冒充运行时开关。

### 安全要求

- 默认禁用原始 HTML，或使用明确、严格的净化策略；
- Mermaid 等模块不得执行文档中的任意脚本；
- 外部链接通过受控系统 API 打开；
- Tauri capabilities 采用最小权限；
- 远程图片和资源加载策略必须可配置并默认保守。

### 阶段门禁

- 模块关闭时不加载其代码块、不创建后台资源；
- 模块开启后仍不进入普通输入同步路径；
- 关闭模块后实例资源可被清理；
- KaTeX 和 Mermaid 各自具有独立压力测试；
- 生成 `docs/checkpoints/phase-5.md`。

---

## 阶段 6：Git 备份与同步模块

### 目标

实现可关闭、异步、不会污染输入主路径的 Git 备份能力。

### 技术策略

- 默认使用用户已安装的系统 Git CLI；
- Git 功能默认关闭；
- 复用系统 SSH、HTTPS Credential Manager、代理、Git LFS 和 hooks；
- Git 命令进入每仓库串行队列，同一仓库同一时间只运行一个任务；
- 文件保存、Git Commit、Git Push 是三个独立动作。

### 调度规则

1. 只有文档保存成功后，才能进入 Git dirty 状态。
2. “最短 30 秒”表示两次自动 checkpoint 的最小间隔，不表示永久运行的 30 秒轮询。
3. 第一次变脏时计算下一次允许执行时间并创建一次性任务。
4. 状态干净后不再创建定时任务。
5. 手动保存可按配置立即触发 checkpoint。
6. Commit 与 Push 分离；Push 在手动同步、较长空闲或独立低频策略下运行。
7. 默认不自动 Pull 到有本地修改的工作区。
8. 不自动强推，不 amend 已经推送的 commit。
9. 建议提供应用专用 `autosave/<device>` 分支模式，避免污染用户正常分支。

### 测试

- 非 Git 目录；
- 干净仓库和脏仓库；
- detached HEAD；
- 合并冲突；
- 无远程仓库；
- Push 失败、断网、认证失败；
- 两个文档同时请求备份；
- 应用退出时队列中有任务；
- 路径包含中文和空格；
- Git 功能关闭时不创建任何 Git 进程和定时器。

### 阶段门禁

- Git 操作绝不阻塞 UI；
- 干净状态和功能关闭状态下无 Git 周期唤醒；
- 失败不会影响本地文件保存；
- 不会静默改写用户分支历史；
- 生成 `docs/checkpoints/phase-6.md`。

---

## 阶段 7：工作区、会话恢复与惰性文件树

### 目标

在不把应用变成 IDE 或知识库数据库的前提下，提供基本工作区体验。

### 实施内容

1. 文件树默认只读取根目录第一层。
2. 用户展开目录时才读取该层内容，不主动递归扫描。
3. 不默认监听整个目录树；需要刷新时优先按用户操作或窗口重新聚焦触发。
4. 若要实时刷新文件树，只监听已展开的有限目录，并设置最大监听数量。
5. 配置使用简单、原子写入的 JSON/TOML；没有查询需求时不引入 SQLite。
6. 会话状态保存：已打开文件、活动标签、光标位置、滚动位置。
7. 状态写入采用失焦或变更后的低频一次性任务，不做高频轮询。
8. 建立崩溃恢复快照，但必须与用户正式 Markdown 文件分离。

### 阶段门禁

- 打开大目录不会自动递归扫描；
- 文件树关闭时无目录后台活动；
- 恢复数据损坏不会阻止应用启动；
- 会话状态写入不影响输入；
- 生成 `docs/checkpoints/phase-7.md`。

## 阶段 8：跨平台硬化与发布门禁

### 目标

完成 macOS、Windows 的真实环境验证和发布准备。

### 实施内容

1. macOS WKWebView 与 Windows WebView2 的字体、选择、滚动和 CSS 差异处理。
2. 中文输入法、快捷键、菜单、文件路径、DPI 缩放测试。
3. 使用系统 Evergreen WebView2，除非离线部署有明确需求，不默认捆绑 Fixed Runtime。
4. 收紧 Tauri capabilities 和 CSP。
5. 完成应用签名、安装包、升级和崩溃处理策略。
6. 默认关闭遥测；如未来加入，必须显式同意且不能高频上报。
7. 在至少一台 macOS 和一台 Windows 真实设备上运行完整性能与功耗矩阵。

### 发布门禁

- 所有数据安全测试通过；
- 所有核心性能门禁通过，或对未通过项有明确降级策略；
- 前台和后台静置无应用自建高频唤醒；
- 连续编辑、滚动、保存、切换文档 2 小时无明显内存泄漏；
- Git、KaTeX、Mermaid 全部关闭时，核心应用仍独立可用；
- Release 构建不包含调试级高频日志；
- 生成最终性能报告和发布检查表。

## 7. 跨阶段测试矩阵

### 7.1 编辑正确性

- 光标在标记前、标记内、内容内、标记后；

### 7.3 资源释放

- 窗口隐藏、恢复、休眠、唤醒。

## 8. 代码质量与依赖规则

1. TypeScript 使用严格模式，避免无边界的 `any`。
2. Rust 错误使用结构化错误类型，不把所有错误简单压成字符串。
3. IPC 请求和响应定义明确的版本与类型。
4. 新增依赖前记录：用途、包体影响、运行时影响、替代方案、是否可懒加载。
5. 不引入同时解决同一问题的多个库。
6. 不对 CodeMirror 内部 DOM 进行未声明的直接操作。
7. 所有异步任务都有取消、revision 校验或过期丢弃机制。
8. 所有缓存都有容量限制和清理入口。
9. 所有后台任务都有可观察的生命周期。
10. 禁止在性能关键路径输出高频日志。

建议持续运行：

```text
npm run check
npm run test
npm run test:e2e
npm run bench:editor
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
```

实际命令可由阶段 0 确定，但之后应保持稳定。
