# 阶段 1 性能基线

- 测量日期：2026-07-14
- 硬件与系统：MacBook Pro（Mac17,8），Apple M5 Pro，macOS 26.5（25F71），arm64
- Release 工具：`xctrace 16.0 (17F113)` Time Profiler，录制 All Processes
- 合成测试：Node 24.16.0、Vitest 4.1.10、jsdom

## 可见范围 Decorations

固定读取文档开头 2,000 字符的可见范围，重复构建 100 次。结果仅衡量 Lezer tree traversal 和 Preview range 生成，不包含浏览器绘制。

| 文档            | 节点访问数 |      p50 |      p95 |      p99 |
| --------------- | ---------: | -------: | -------: | -------: |
| basic（32 KiB） |        562 | 0.097 ms | 0.235 ms | 0.832 ms |
| medium（1 MiB） |        562 | 0.072 ms | 0.096 ms | 0.141 ms |

两种文档访问节点数相同，说明装饰构建受可见范围约束，没有随不可见全文长度线性增长。

## 完整编辑器合成更新

在 jsdom 中挂载完整 `EditorView`，在文档末尾连续 dispatch 100 次单字符输入。该结果覆盖 CodeMirror transaction、增量解析、Live Preview plugin 更新和同步 DOM 更新，但不等同于真实按键到屏幕帧延迟。

| 文档            |      p50 |      p95 |      p99 |          门禁 |
| --------------- | -------: | -------: | -------: | ------------: |
| basic（32 KiB） | 0.317 ms | 1.016 ms | 4.374 ms |    p95 < 8 ms |
| medium（1 MiB） | 0.257 ms | 0.429 ms | 3.599 ms | p95 < 16.7 ms |

阶段 0 的真实浏览器输入到帧基线中 basic p95 为 6 ms、medium p95 为 8 ms；本阶段新增的可见装饰构建 p95 低于 0.24 ms。由于本轮浏览器控制运行时不可用，不能把上述 jsdom 数据冒充新的浏览器帧测量。Release 原生 UI 已验证连续输入和装饰更新无明显停顿，后续浏览器或原生帧采样恢复后仍应补充端到端数据。

## Release 空闲 CPU

每组均录制约 60.59 秒，并在录制前后通过 macOS accessibility 确认前台与焦点状态。

| 场景             | 主进程 | WebContent |    GPU | 进程组 CPU 时间 | 平均 CPU |
| ---------------- | -----: | ---------: | -----: | --------------: | -------: |
| 前台未聚焦编辑器 |   9 ms |       3 ms |   0 ms |           12 ms |   0.020% |
| 前台聚焦编辑器   | 483 ms |     504 ms | 374 ms |        1,361 ms |   2.246% |

未聚焦结果通过 ≤0.5% 分层门禁；聚焦结果没有高于阶段 0 的 2.51% WebKit 平台基线。Live Preview 没有引入空闲重复构建或持续动画。

## 构建产物

生产前端输出为 630.47 kB minified / 217.88 kB gzip；相比阶段 0 优化后的 626.71 kB / 216.66 kB，增量约 3.76 kB / 1.22 kB。Vite 的单 chunk 超过 500 kB 警告仍存在。
